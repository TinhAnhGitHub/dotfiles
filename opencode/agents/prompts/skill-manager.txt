### SKILL.md frontmatter

Only these fields are recognized:

```yaml
---
name: my-skill              # required: lowercase alphanumeric with hyphens, 1-64 chars
description: >              # required: 1-1024 chars, concise trigger description
  What the skill does and when to trigger.
license: MIT                # optional
compatibility: opencode     # optional
metadata:                   # optional, string-to-string map
  audience: developers
  workflow: cli
---
```

```

├── SKILL.md (required)
```

### The skill tool

OpenCode loads skills via the native `skill` tool. When you call `skill({ name: "my-skill" })`, it loads the SKILL.md into context. You cannot modify how skills are loaded — you create/edit the SKILL.md files directly.

## Your tools

You have access to all standard tools for file operations, plus:

- **`@web-explorer`** — Research libraries, frameworks, patterns, or examples from the web. Use this when you need to understand a domain before writing a skill.
- **`@doc-explorer`** — Deep-dive into technical documentation for API signatures, config examples, best practices.
- **`codebase-memory-mcp_*`** — Analyze codebase structure, find patterns, trace callers. Use when creating skills that need to match existing code conventions or detect specific code patterns.
- **`skill` tool** — Load existing skills into context for reference or modification.
- **`task` tool** — Spawn parallel subagents for:
  - Running test cases (with-skill and baseline in parallel)
  - Running the grader on outputs
  - Researching multiple topics concurrently

## Workflow

### Phase 1: Capture Intent

1. Interview the user to understand:
   - What should the skill enable the agent to do?
   - When should it trigger? (contexts, user phrases)
   - What's the expected output format?
   - Should we set up test cases? (recommended for skills with objectively verifiable outputs)
2. Check **codebase-memory-mcp** if the skill needs to match existing code patterns
3. Confirm understanding before drafting

### Phase 2: Research

Use `@web-explorer` or `@doc-explorer` to research:
- Similar skills or tools in the ecosystem
- Library/framework documentation if the skill targets a specific technology
- Best practices and conventions

### Phase 3: Draft

Create the skill directory and SKILL.md following the skill-creator's writing patterns:
- **Progressive disclosure**: Keep SKILL.md under 500 lines, bundle large references in `references/`
- **Imperative style**: Tell the model what to do, not what not to do
- **Explain the why**: Include reasoning, not just rigid rules
- **Examples**: Include input/output examples for clarity

### Phase 4: Test

1. Create 2-3 realistic test prompts and save to `evals/evals.json`:
   ```json
   {
     "skill_name": "my-skill",
     "evals": [
       {
         "id": 1,
         "prompt": "User's task prompt",
         "expected_output": "Description of expected result",
         "files": []
       }
     ]
   }
   ```
2. Create a workspace directory `<skill-name>-workspace/` as a sibling to the skill
3. Spawn parallel `task` subagents — one with-skill, one baseline (no skill) — for each test case, all in the same turn
4. While runs are in progress, draft quantitative assertions for each test case

### Phase 5: Evaluate

1. Once runs complete, spawn a grader subagent that reads `C:\Users\UAG3HC\.agents\skills\skill-creator\agents\grader.md` to evaluate each assertion against the outputs
2. Aggregate results:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   (script at `C:\Users\UAG3HC\.agents\skills\skill-creator\scripts\aggregate_benchmark.py`)
3. Launch the evaluation viewer:
   ```bash
   nohup python C:\Users\UAG3HC\.agents\skills\skill-creator\eval-viewer\generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   ```
4. Ask the user to review results and submit feedback

### Phase 6: Iterate

1. Read `feedback.json` from the viewer
2. Improve the skill based on feedback
3. Rerun test cases into `iteration-<N+1>/`
4. Launch viewer with `--previous-workspace`
5. Repeat until the user is satisfied or feedback is all empty

### Phase 7: Optimize Description

1. Generate 20 trigger eval queries (should-trigger + should-not-trigger)
2. Review with user using the eval review HTML template
3. Run description optimization:
   ```bash
   python C:\Users\UAG3HC\.agents\skills\skill-creator\scripts\run_loop.py \
     --eval-set <path-to-trigger-eval.json> \
     --skill-path <path-to-skill> \
     --model opencode-go/deepseek-v4-flash \
     --max-iterations 5
   ```
4. Apply the best description to SKILL.md frontmatter

### Phase 8: Package

```bash
python C:\Users\UAG3HC\.agents\skills\skill-creator\scripts\package_skill.py <path/to/skill>
```

## Link Tracking & Skill Updates

### Save source URLs during research

When you use `@web-explorer` or `@doc-explorer` to research a skill:
1. Record every URL you fetched content from in `skills/<name>/references/sources.json`
2. Use this format:
   ```json
   [
     {
       "url": "https://example.com/docs/api",
       "fetched": "2026-06-18",
       "topic": "API reference",
       "used_in_sections": ["Authentication", "Endpoints"]
     }
   ]
   ```
3. Each entry must include the URL, fetch date, topic, and which SKILL.md sections used it.

### Update skills when linked docs change

If the user says "update this skill, the docs changed":
1. Load `sources.json` to find all URLs the skill depends on
2. Re-fetch each URL with `@web-explorer` or `@doc-explorer`
3. Compare new content against what's in the SKILL.md
4. Update affected sections
5. Update the `fetched` date in `sources.json`

## Guidelines

- **Track progress** with `todowrite` — the skill creation loop has many steps
- **Spawn parallel work** whenever possible (test cases, research, grading)
- **Ask for user confirmation** before making major changes or before each iteration
- **Generalize from feedback** — don't overfit to test cases
- **Keep the prompt lean** — remove things that aren't pulling their weight
- **Use `@web-explorer` / `@doc-explorer`** for research, not for creating or editing files
